## 주요 신규 기능 & 실전 활용

### `/tui` — 깜빡임 없는 풀스크린 터미널 모드 (v2.1.110)

v2.1.110에서 `/tui` 명령과 `tui` 설정이 추가되었습니다. 기존 터미널 렌더링에서 발생하던 깜빡임(flicker) 없이 풀스크린으로 Claude Code를 사용할 수 있습니다. 장시간 세션에서 출력이 빠르게 스크롤될 때 시각적 안정성이 크게 개선됩니다.

```bash
# 풀스크린 TUI 모드 전환
/tui

# settings.json에서 기본 활성화
# { "tui": true }

# 풀스크린 모드에서 자동 스크롤 비활성화 (긴 출력 검토 시)
# { "autoScrollEnabled": false }
```

`Ctrl+O`가 일반/verbose 트랜스크립트 전환 전용으로 단순화되었고, 외부 에디터에서 Claude의 마지막 응답을 주석 컨텍스트로 표시하는 옵션도 추가되었습니다. 대량의 코드 변경을 리뷰하거나 장시간 에이전트 작업을 모니터링할 때 유용합니다.

[GitHub v2.1.110](https://github.com/anthropics/claude-code/releases)

### Push Notification — 모바일 알림 도구 (v2.1.110)

v2.1.110에서 Push Notification 도구가 추가되었습니다. 장시간 작업이 완료되거나 사용자 개입이 필요할 때 모바일 기기로 알림을 받을 수 있습니다.

기존에는 터미널을 지속적으로 모니터링하거나 서드파티 알림 도구를 연결해야 했지만, 이제 Claude Code가 직접 푸시 알림을 전송합니다. Routines나 `/loop`과 결합하면 클라우드 또는 로컬에서 실행 중인 자동화 작업의 상태를 이동 중에도 확인할 수 있습니다.

## 개발자 워크플로우 팁

### Claude Code + Codex 하이브리드 워크플로우 — 속도와 정확성의 최적 조합

GeekNews에서 39포인트를 받은 "Claude Code(~100시간) vs. Codex(~20시간) 비교"에서, MAG7 출신 14년차 시니어 엔지니어가 대규모 Python/TypeScript 프로젝트에서 두 도구를 비교했습니다.

**핵심 발견**:
- **Claude Code (Opus 4.6)**: 빠르고 인터랙티브하지만, 지시를 무시하거나 작업을 미완성으로 남기는 경향. "충분한 사전 검토 없이 구현을 밀어붙인다"
- **Codex (GPT-5.4)**: 3~4배 느리지만 신중하고 의도적. AGENTS.md를 엄격히 준수하며 자발적 리팩토링 수행

**커뮤니티 합의 — 하이브리드 워크플로우**:
```
Claude Code plan 모드 → Sonnet 구현 → Codex 리뷰 → 반복
```

단일 도구에 올인하기보다 각 도구의 강점에 맞게 역할을 분담하는 패턴이 실전에서 가장 효과적이라는 결론입니다. Claude의 속도로 빠른 프로토타이핑을 수행하고, Codex의 꼼꼼함으로 리뷰와 품질 검증을 담당하는 분업입니다.

[GeekNews](https://news.hada.io/topic?id=28538)

### `--resume`으로 스케줄 작업까지 복원하기 (v2.1.110)

v2.1.110에서 `--resume`과 `--continue`가 아직 만료되지 않은 스케줄 작업을 자동 복원합니다. 이전에는 세션을 재개해도 `/loop`이나 CronCreate로 설정한 스케줄이 사라졌지만, 이제 세션 복구 시 활성 스케줄도 함께 되살아납니다.

CI 빌드 감시, 로그 폴링 같은 장기 모니터링을 `/loop`으로 설정한 뒤 세션을 닫았다가 재개해도 스케줄이 유지됩니다. `/recap`과 조합하면 "어디까지 했는지 + 무엇을 감시하고 있었는지" 모두 한 번에 복원할 수 있습니다.

### Claude 성능 저하 논란 — effort 기본값 "medium" 하향 조정 파문

Fortune 보도에 따르면, Anthropic이 3월 초 Claude의 기본 effort 수준을 "high"에서 "medium"으로 변경한 것이 광범위한 사용자 불만을 야기하고 있습니다.

**사용자 불만**:
- 지시 준수 실패 증가, 부적절한 단축 경로 선택, 복잡한 작업에서 실수 빈번
- "연구 우선(research-first)" 접근에서 "편집 우선(edit-first)" 접근으로 전환
- "복잡한 엔지니어링 작업에 사용할 수 없게 되었다"는 분석가 평가

**Anthropic 대응** (Boris Cherny, Claude Code 리드):
- 변경은 changelog에 기록되었고, 과도한 토큰 소비 피드백에 대응한 것
- 추론 능력 자체는 감소하지 않았으며, 사용자에게 덜 보이게 된 것뿐
- 엔터프라이즈 사용자에게는 향후 high effort를 기본값으로 전환 예정

```bash
# 즉시 해결: effort 레벨 수동 지정
claude --effort high
```
