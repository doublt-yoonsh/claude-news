## 개발자 워크플로우 팁

### "플로저블 코드" 함정 — 수용 기준을 먼저 정의하라

"LLM은 올바른 코드를 작성하지 않는다. 그럴듯한 코드를 작성할 뿐이다"라는 분석 글이 화제입니다. 저자가 SQLite를 Rust로 재구현하도록 LLM에 요청한 결과, **테스트를 통과하고 컴파일에 성공했지만 기본 키 조회 속도가 20,000배 느린** 구현이 나왔습니다.

**발견된 치명적 버그:**
- `INTEGER PRIMARY KEY` 구문을 인식하지 못해 O(log n) 인덱스 대신 전체 테이블 스캔 수행
- 매 쿼리마다 fsync를 호출하여 SQLite 대비 78배의 오버헤드 발생

이 문제의 근본 원인은 LLM의 **"아첨 패턴(sycophancy)"** — 정말로 올바른 코드보다 올바르게 보이는 코드를 생성하는 경향입니다. Claude Code를 사용할 때 실천할 수 있는 교훈:

1. **코드 생성 전에 성능 기준을 명시**: "기존 대비 동등한 성능" 같은 구체적 기준 포함
2. **테스트 통과 ≠ 정확성**: 기능 테스트 외에 성능 벤치마크, 엣지 케이스 테스트 추가
3. **아키텍처 제약 조건 명시**: 사용해야 할 알고리즘, 자료구조, I/O 패턴을 프롬프트에 포함

### Claude가 바꾸는 소프트웨어 제작의 문턱

Washington Post가 Claude Cowork을 실전 테스트하여 "20초 타이핑 후 수 분 만에 미디어 추적 웹사이트를 완성"한 과정을 인터랙티브 기사로 공개했습니다. 한편, Hacker News에서는 60세 개발자가 "Active Server Pages와 VB6 시절의 열정이 Claude Code 덕분에 다시 불타오르고 있다"는 글이 큰 반향을 일으키고 있습니다.

**주목할 트렌드:**
- 소설가 Eliot Peper는 코딩 경험 없이 Claude로 서핑 예보 앱 "Dialed"을 만들어 App Store에 100명 이상의 사용자를 확보
- Simon Willison: "Claude Code 출시 초기에는 첫 시도에서 작동하면 항상 놀라웠다. 이제는 그것이 당연하다"
- Claude Cowork(비개발자용 GUI)과 Claude Code(개발자용 CLI)의 역할 분리가 명확해지는 추세

개발자에게 시사하는 바: Claude Code의 진정한 가치는 단순한 코드 생성이 아니라, **코드베이스 이해와 깊은 통합 능력**에 있습니다. 비개발자 도구가 "무엇을 만들 것인가"를 민주화한다면, Claude Code는 "어떻게 잘 만들 것인가"에서 개발자의 생산성을 극대화합니다.

Google이 Microsoft에 이어 Claude를 비방위 고객에게 계속 제공하겠다고 공식 확인했습니다. Amazon(AWS)도 동일한 입장을 밝혀, **미국 3대 클라우드 업체 모두 Pentagon 공급망 리스크 지정이 방위 계약에만 국한**된다는 입장을 표명했습니다.

이로써 엔터프라이즈 고객이 우려할 수 있었던 **클라우드 플랫폼을 통한 Claude 접근 차단 시나리오는 사실상 해소**되었습니다. 다만 방위 계약에 참여하는 기업은 여전히 Claude 미사용 인증이 필요합니다.
